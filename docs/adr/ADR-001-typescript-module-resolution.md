# ADR-001: TypeScript Module Resolution Strategy

## Status
Accepted

## Context

TypeScript 5.0+ introduced the `Bundler` module resolution mode alongside the existing `Node16`/`NodeNext` modes. Our monorepo contains:

1. **Node.js runtime packages** - `packages/db`, `apps/api`, `apps/graphql`
2. **Frontend apps using Vite** - `apps/web`, `apps/admin`
3. **Shared validation package** - `packages/validation` (generated by `prisma-zod-generator`)

Each has different requirements for module resolution.

## Decision

We use a **hybrid module resolution strategy**:

### Base Configuration (`tsconfig.base.json`)
```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext"
  }
}
```

### Per-Package Overrides

| Package | moduleResolution | Reason |
|---------|------------------|--------|
| `packages/db` | NodeNext (inherited) | Direct Node.js runtime, Prisma client |
| `packages/validation` | **Bundler** | Generated code lacks `.js` extensions |
| `apps/api` | NodeNext (inherited) | Node.js HTTP server |
| `apps/graphql` | NodeNext (inherited) | Node.js GraphQL server |
| `apps/web` | **Bundler** | Vite handles resolution |
| `apps/admin` | **Bundler** | Vite handles resolution |

## Module Resolution Modes Explained

### NodeNext (Node16)
- **Strict ESM compliance** for Node.js runtime
- **Requires `.js` extensions** in relative imports (`import { x } from './foo.js'`)
- **Respects `package.json` exports** field strictly
- **Use when**: Code runs directly in Node.js

### Bundler
- **Designed for bundlers** (Vite, webpack, esbuild, Rollup)
- **Extensions optional** in imports (`import { x } from './foo'`)
- **Flexible `package.json` resolution** - bundler handles it
- **Use when**: Code is processed by a bundler before runtime

## Why Validation Uses Bundler

The `prisma-zod-generator` produces TypeScript files with imports like:

```typescript
// Generated by prisma-zod-generator (no .js extension)
import { ProductCategorySchema } from '../../enums/ProductCategory.schema';
```

Using `NodeNext` would require post-processing to add `.js` extensions. Instead, we:
1. Use `moduleResolution: "Bundler"` for the validation package
2. Consumers (api, graphql) that use `NodeNext` import from the package entry point
3. The validation package's `dist/` output is consumed, not the source

## Import Pattern by Package Type

### Node.js Packages (NodeNext)
```typescript
// Must include .js extension for local imports
import { prisma } from './client.js';
import { handler } from '../routes/users.js';

// Package imports don't need extensions
import { ProductValidationSchema } from '@octant/validation';
```

### Bundled Packages (Bundler)
```typescript
// Extensions optional - bundler resolves
import { ProductInputSchema } from './generated/schemas/variants/input/index';
import { ProductCategorySchema } from '../../enums/ProductCategory.schema';
```

## Validation Package Include Strategy

The validation package uses explicit `include` to avoid compiling unused generated schemas:

```json
{
  "include": [
    "src/index.ts",
    "src/generated/schemas/variants/**/*",
    "src/generated/schemas/enums/**/*"
  ]
}
```

This excludes the CRUD operation schemas (which have `@prisma/client` dependencies) while including only the Prisma-free variant and enum schemas.

## Consequences

### Positive
- Node.js apps get strict ESM compliance and better error messages
- Frontend apps work seamlessly with Vite's resolution
- Generated code works without post-processing
- Clear separation of concerns

### Negative
- Mixed resolution modes require understanding which mode each package uses
- Contributors must know to use `.js` extensions in Node.js packages
- IDE support varies (VSCode handles both well)

### Mitigations
- Document the pattern in this ADR
- ESLint can enforce extension usage in NodeNext packages
- TypeScript errors clearly indicate missing extensions
