# ADR-XXX: Vote Prover

## Status
Draft

## Context

Collects votes on L2 but distributes matching on mainnet. Bridging vote results between chains requires a trustless mechanism that:

1. Does not trust a relayer — mainnet must verify L2 vote outcomes cryptographically
2. Scales with chain count, not vote count — one proof per L2 regardless of how many users voted
3. Keeps gas costs constant — verifying a proof for 10,000 votes costs the same as for 10

Storage proofs satisfy all three. The L2 contract maintains a merkle root over all vote leaves in a single storage slot. A ZK circuit proves the value of that slot via MPT verification, and a Solidity verifier checks the Groth16 proof on mainnet.

### Alternatives considered

| Approach | Why not |
|----------|---------|
| Coprocessor network (Brevis) | MPT verification delegated off-chain; external infrastructure dependency |
| Merkle proof per vote | O(n) proofs — one per vote, doesn't scale |
| Optimistic bridging | Challenge window (hours/days); trust assumptions on watchers |
| Raw gnark (no framework) | Must implement EVM MPT verification in circuit constraints manually |

## Decision

### Architecture

```
L2                                 Off-chain                     Mainnet
┌───────────────────┐     ┌─────────────────────┐     ┌──────────────────────┐
│  VotingContract   │     │     vote-prover      │     │   VoteVerifier       │
│                   │     │                      │     │                      │
│  vote()           │     │  1. eth_getProof     │     │   verifyVoteRoot(    │
│  voteMerkleRoot   │ RPC │  2. Build MPT input  │     │     proof,           │
│  (slot 1)         │────>│  3. Prove (Pico)     │────>│     publicValues)    │
│                   │     │  4. Groth16 wrap     │     │                      │
│                   │     │                      │     │   → Groth16Verifier  │
└───────────────────┘     └─────────────────────┘     └──────────────────────┘
```

### VotingContract storage layout

```
Slot 0: currentEpoch (uint256)       — epoch counter, starts at 1
Slot 1: voteMerkleRoot (bytes32)     — the slot proven by the ZK circuit
Slot 2: voteCount (uint256)          — total votes cast
Slot 3: leaves.length (bytes32[])    — dynamic array length; data starts at keccak256(3)
```

Each vote leaf is `keccak256(abi.encodePacked(voter, projectId, amount, epoch))`. The merkle tree is binary, padded to the next power of two, recomputed in full on each `vote()` call.

### Circuit design

The guest program runs inside Pico's RISC-V zkVM and performs two MPT proof verifications using `alloy-trie`:

1. **Account proof** — verifies the contract account exists in the state trie by walking MPT nodes from `state_root` to the RLP-encoded account
2. **Storage proof** — verifies the storage value exists in the account's storage trie by walking MPT nodes from `storage_hash` to the RLP-encoded value

```rust
verify_proof(state_root, keccak256(address), Some(rlp(account)), &account_proof)?;
verify_proof(storage_root, keccak256(slot), rlp_or_none(value), &storage_proof)?;
```

For zero-valued storage slots (absent keys in the MPT), `verify_proof` is called with `None`, correctly handling exclusion proofs.

Public outputs committed by the circuit:

| Field | Size | Description |
|-------|------|-------------|
| `state_root` | 32 bytes | Block state root the proof is anchored to |
| `contract_address` | 20 bytes | L2 VotingContract address |
| `storage_slot` | 32 bytes | Slot index (1 = voteMerkleRoot) |
| `storage_value` | 32 bytes | The verified voteMerkleRoot value |

### On-chain verification

Two contracts on mainnet:

- **Groth16Verifier** — generated by gnark, verification key baked in as constants. BN254 pairing-based verification (~230K gas). Deterministic per ELF binary.
- **VoteVerifier** — takes a Groth16 proof and the raw public values bytes, reconstructs the two Groth16 public inputs, calls Groth16Verifier, then decodes and emits the verified values.

The Groth16 public inputs follow Pico's commitment scheme (identical to SP1):

| Index | Name | Derivation |
|-------|------|------------|
| `inputs[0]` | `vkeyHash` | Hash of the RISC-V program verification key (deterministic per ELF) |
| `inputs[1]` | `committedValuesDigest` | `sha256(publicValues) & ((1 << 253) - 1)` |

The 253-bit mask ensures values fit within the BN254 scalar field. VoteVerifier stores the expected `vkeyHash` as an immutable, ensuring only proofs from the correct circuit are accepted.

### Trust assumptions

- MPT proofs are verified inside the zkVM — a dishonest prover cannot fabricate storage values
- The proof is anchored to a specific block's state root, which must be validated against a trusted source of L2 block headers on L1 (e.g., the L2's canonical bridge or a light client)
- The VotingContract code must be correct — the proof verifies storage state, not contract logic
- Constant proof size (~256 bytes) and constant verification gas (~230K) regardless of vote count

## Consequences

### Positive
- Fully self-contained — no dependency on Brevis coprocessor network or any external service
- True MPT verification — storage values verified against state root inside the zkVM
- On-chain verifiable — Groth16 proofs verified by a Solidity contract (~230K gas)
- Constant verification gas independent of vote count
- Scales to N chains — one proof per L2
- Simpler dependency tree — only `pico-sdk`, `alloy-trie`, `alloy-primitives`

### Negative
- Rust nightly dependency — Pico requires `nightly-2025-08-04`
- Per-circuit trusted setup — Groth16 needs new setup when circuit changes; mitigated by simple circuit that rarely changes
- O(n) merkle rebuild — `_computeRoot()` recomputes the full tree on every vote; an incremental tree would reduce to O(log n)
